## Why not promoted

已抓到 closed PR #42495 与更新的 open PR #45981。#42495 用 per-load cache key 修 stale reuse，但靠进程内计数器与 path fallback；#45981 改为 loader-effective 的 content-derived `lora_cache_key`，更贴近根因：选中的 weight file、canonicalized config、`is_3d_lora_weight` 与 tensorizer config 一起进 hash，并由 serving load path 与 `process_inputs` backstop 共用 `ensure_lora_cache_key` 填充；block-hash 也拆成 `(lora, name) + (lora_identity, id)` 两层 extra keys。结合当前 live patch，这条线其实已经形成 identity 计算 (`vllm/lora/cache_identity.py`) -> request 传播 (`serving.py` + `input_processor.py`) -> block-hash 消费 (`kv_cache_utils.py`) 的三层闭环。测试除 same-name different-path、same-path different-cache-key、same-content same-key、config/weight 变化改 key 外，还覆盖了 safetensors 优先级、length framing collision、relative path path-only fallback。PR body 同时明确把 claim 缩到本地可读 source：HF Hub、relative/remote path、shared-nothing/external KV、TOCTOU 与 old EngineCore peer/rolling-upgrade 仍是剩余边界；截至 2026-06-21，`#45981` 仍只有作者正文与 bot 活动，reviewers 仍处于 awaiting requested review，尚无 substantive maintainer review。

## Evidence summary

新增 insight：runtime same-name reload 的本地 prefix-cache key 不能只看 `lora_name`、`lora_int_id` 或 path；真正需要的是 loader-effective 的内容派生 adapter identity，而且 `#45981` 已把它做成跨进程可复现的 content hash，而不是 per-process counter。与 external KV 线相比，这条不仅 patch 结构、identity 定义、fill 点与测试矩阵更成型，连 identity 计算、request 传播、block-hash 消费三层闭环也已经成型；当前剩下的主要不是 patch correctness 争议，而是作者已承认的 source/部署边界与 upstream 接受度。
