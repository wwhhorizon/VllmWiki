## Why not promoted

`#38991` 自身仍 open/0 comments，且截至 2026-06-21 仍查不到任何直接引用 `#38991` 的 PR，也没有 2026-06-20 之后的新 maintainer/commentary 进展；但 issue body 已给出强 unified-memory 证据链：shared numpy buffer view、`BaseModelLoader.load_model()` 下 CPU->CUDA `copy_()` 异步、single sync 与 iterator-side `clone()` 都能消除 corruption。同宗 issue `#43163` 与 merged PR `#43464` 又把这条机制从平台特例推进成上游 accepted correctness fix：对 internal reusable buffer loader，应在 `runai_safetensors_weights_iterator` 边界直接 `yield name, tensor.clone()`，且测试会把 `RUNAI_STREAMER_MEMORY_LIMIT=0` 压到内部 buffer 复用状态，再验证 yielded tensors 不共享 data pointer。结合当前 main 代码路径，`BaseModelLoader.load_model()` 仍在 CUDA 参数上执行 load，而 `runai_safetensors_weights_iterator` 已在 yield 前 clone；`AutoWeightsLoader._load_param()` 会立刻把 tensor 交给 `default_weight_loader` 的 `param.data.copy_()` 消费，因此原始 shared-buffer alias 链在默认路径上已经被机械切断。后续 closed issue `#44430` 与 merged PR `#44645` 进一步说明：clone 的 host-RAM 回归应由 eager materialization 的模型 loader 改成 streaming 来吸收，而不是撤回 correctness fix；但这条 landed 证据直接覆盖的是 Llama4 weight-loading 路径，应作为 family-level 样例而不是 blanket closure；reporter 已在 2026-06-15 明确确认修复。当前仍缺 direct `Fixes #38991`、缺 maintainer closure，也缺 exact unified-memory + `BaseModelLoader.load_model()` async copy 路径的正式 regression test。

## Evidence summary

Weight loading order/lifetime corruption 与 deterministic quant loading 相关；本轮新增 insight：这条线现在已经可以稳定拆成两层 closure: iterator-side `clone()` 是 correctness fix，loader streaming 是 memory/perf 收口，而且两层都有更硬的验证锚点：`#43464` 强制 buffer reuse 的 data-pointer test，`#44645` 的 reporter closure。再结合当前 main 代码路径，默认加载链上的 shared-buffer alias 已被 iterator-side `clone()` 机械切断，因此 `#38991` 继续 defer 的原因更像是 proof gap 而不是 mechanism gap；但 `#44645` 的 streaming closure 需要保持窄范围，先写成 Llama4 landed 样例，而不是所有模型路径都已统一 streaming 化。
