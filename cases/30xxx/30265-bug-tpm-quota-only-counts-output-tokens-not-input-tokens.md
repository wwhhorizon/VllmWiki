### Issue 正文摘录

### Your current environment Environment LiteLLM version: v1.80.0 ### 🐛 Describe the bug When testing the TPM (Tokens Per Minute) quota system in LiteLLM, I discovered that it only counts generated tokens (output/completion tokens) and does not account for input tokens (prompt tokens). This appears to be a bug or design flaw in the quota enforcement system. Expected Behavior The TPM quota should count both: Input tokens (tokens in the user's prompt) Output tokens (tokens generated by the model) This is the standard behavior for token-based rate limiting, as both types of tokens consume API resources and incur costs. Actual Behavior The TPM quota currently only counts output tokens. This means: Users can send extremely large prompts without being rate-limited Only the generated response tokens count toward the quota The quota doesn't accurately reflect actual API usage ## Why This Is Problematic Easy bypass: Users can circumvent rate limits by sending large prompts with short completion requests Inaccurate cost tracking: Input tokens represent real costs that aren't being tracked Incomplete usage monitoring: The quota doesn't reflect true API consumption Security concern: Could be...
