## Why not promoted

PR open/unmerged（有 merge conflict，`needs-rebase`）。gemini-code-assist[bot] 在 `_prefix_cache_aligned_split` 中发现两个 correctness gap：
1. `num_output_tokens > 0` 的 early return 会漏掉 resumed (preempted) request 的 prompt re-prefill。
2. `remainder == 0` 的 early return 会漏掉 block-aligned prompt 的 final-token scheduler 约束（`max_cache_hit_length = request.num_tokens - 1`），导致 32-token prompt + block_size=16 时 miss-M=32 vs hit-M=16 不一致。
建议替换为 `num_computed_tokens >= request.num_prompt_tokens` 和 `((num_prompt_tokens - 1) // self.block_size) * self.block_size`。作者尚未回复，也无人 review。

## Evidence summary

PR body 和 changed files 明确实现 deterministic prefix caching（4 files, +172/-0）。Root cause：cache-miss prefill 全量 N tokens GEMM (M=N) 与 cache-hit suffix GEMM (M=N%block_size) 产生不同 M 维度，触发 ROCm Tensile 不同 tiling 和 fp32 累加顺序，bf16 非结合性将 1 ULP 放大至完全不同的 argmax。Credit: @AndreasKaratzas (issue #33123)。

Core logic：`_prefix_cache_aligned_split()` 在 scheduler 中强制 cache-miss prefill 停在最后一个 block boundary，使 suffix GEMM 与 cache-hit 共享相同 M 维度。两个 call site 分别处理首次调度和排队请求的第二个调度循环。

E2E test：`tests/v1/e2e/test_deterministic_prefix_caching.py`（3 tests，Qwen/Qwen3-0.6B，GPU required）。
