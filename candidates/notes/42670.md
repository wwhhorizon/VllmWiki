## Evidence summary

**根因**：FlashInfer attention 的 split-KV partition 是动态 batch size 的函数，同一 query row 在 m=1 和 m=2 下 attention output 不同。CUTLASS FP4 MoE 的 `supports_batch_invariance()` 继承基类 `False`，导致即使设置 `VLLM_BATCH_INVARIANT=1`，selector/oracle 也拒绝该组合。

**Fix**（2 files, +26/−0）：`FlashInferBackend.supports_batch_invariance() → True` + `CutlassExpertsFp4._supports_batch_invariance() → True`。两个 code path 原本已实现 batch-invariant 逻辑，只是被 dead gate 挡住。

**再现**：MiniMax-M2.7-NVFP4 + TP=2 + concurrency=2 + 同 prompt 4 → baseline 2/2 bad（重复循环），PR+env → 0/2 bad，两 req 的 sha256 完全一致 `224a0faefe...`。分歧始于 layer 0 self_attn output（sum_diff≈0.27 on row magnitude ~1080）。

**Honest framing**：MoE 量化（NVFP4, ModelOpt v0.43.0 标准 recipe）使 greedy top-k expert routing 在 temperature=0 时近乎平局，任何噪声源都可翻转。Expert scale 分布有 heavy tails（layer 5: max≈138x median）。

**Scope 排除**：`FlashInferExperts`（moe_backend=flashinfer_cutlass）被有意排除 —— author 测试发现它产生 deterministic bad output（两 req bit-exact 相等但陷入 repetition attractor）。
